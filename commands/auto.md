# Auto Command

Run the marketing pack-local Aaron workflow (SEO/GEO + influencer + ad/ROAS + email/SEND + launch/RAMP + social/ECHO) implied by a natural-language goal, at the smallest safe depth — the natural-language front door to `/aaron-marketing:seo-geo`, `/aaron-marketing:influencer`, `/aaron-marketing:ad`, `/aaron-marketing:email`, `/aaron-marketing:launch`, and `/aaron-marketing:social`. Add `--deep` for an exhaustive, phase-gated, maximum-rigor run.

## Route

- resolver across all 103 skills (16 SEO/GEO + 16 influencer + 16 paid + 16 email + 16 launch + 16 social + 7 protocol): SEO/GEO through `/aaron-marketing:seo-geo` (--mode research|create|audit|track), influencer through `/aaron-marketing:influencer`, paid through `/aaron-marketing:ad`, email through `/aaron-marketing:email` (--phase setup|engage|nurture|deliver), launch through `/aaron-marketing:launch` (--phase research|assemble|mobilize|prove), organic social through `/aaron-marketing:social` (--phase explore|craft|host|observe); or describe any goal here and it routes to the right skill
- **Launch disambiguation**: "launch a product with creators" (or any launch goal centered on creator partnerships) routes to campaign-planner via `/aaron-marketing:influencer`; every other product-launch goal (positioning, launch plan, launch day, Product Hunt / press / app-store release, post-launch retro) routes to `/aaron-marketing:launch`. An email "launch campaign" (a broadcast) stays with `/aaron-marketing:email`; a paid "campaign launch" (turning on ads) stays with `/aaron-marketing:ad`.
- **Social disambiguation — the three-way "post this" split**: "post this / plan our posts / what should we post this week" (the always-on brand calendar and net-new platform-native packages) routes to social-calendar-builder / social-creative-builder via `/aaron-marketing:social`; "boost this post / turn this video into ads / repurpose this asset" routes to content-amplifier via `/aaron-marketing:influencer` (paid execution in `/aaron-marketing:ad`); "launch-day community posts / submit to Product Hunt / Show HN" routes to community-launch-runner via `/aaron-marketing:launch`. Comment/DM/mention triage and UGC permissions route to engagement-inbox-manager; brand/community listening, share of voice, and dark-social attribution route to `/aaron-marketing:social --phase observe`; creator deliverable review stays with content-reviewer via `/aaron-marketing:influencer`.

## Rules

- Follow the routing contract in [references/aaron-product-api-contract.md](../references/aaron-product-api-contract.md); use [references/auto-routing-scenarios.md](../references/auto-routing-scenarios.md) as the scenario library (both under the plugin root — resolve against `${CLAUDE_PLUGIN_ROOT}` when installed). Natural-language auto-invocation is host-dependent unless `/aaron-marketing:auto` is explicit. For influencer goals, route to the matching influencer skill by name or `/aaron-marketing:influencer` phase; for paid-ads goals, route to the matching ROAS-phase skill by name or `/aaron-marketing:ad` phase; for email goals, route to the matching SEND-phase skill by name or `/aaron-marketing:email` phase; for product-launch goals, route to the matching RAMP-phase skill by name or `/aaron-marketing:launch` phase; for organic-social goals, route to the matching ECHO-phase skill by name or `/aaron-marketing:social` phase (`/aaron-marketing:seo-geo` covers SEO/GEO only). Only clearly non-marketing work stops with a pack-boundary note — influencer, paid, email, launch, and social goals are in scope and are routed, never declined.
- **Memory-management exemption**: memory-management lifecycle operations (query, archive, restore, purge) operate on user-owned project memory regardless of topical scope and are NOT subject to the pack-boundary check; safety is enforced by per-operation user confirmation.
- Algorithm: parse the goal -> assign the closest scenario family -> attach risk gates -> choose the smallest `/aaron-marketing:seo-geo` mode (research / create / audit / track), `/aaron-marketing:influencer` phase, `/aaron-marketing:ad` phase, `/aaron-marketing:email` phase, `/aaron-marketing:launch` phase, `/aaron-marketing:social` phase, or chain that completes the outcome -> ask only for blocking inputs -> continue to the natural stopping point.
- Ambiguity rule: with an object (URL/domain/topic/draft) but no clear goal, run lightweight triage and pick the safest useful starting chain; with no actionable object and no outcome, ask one concise blocking question.
- Default to a single seo-geo mode / influencer phase / ad phase / email phase / launch phase / social phase when it completes the goal; otherwise chain (for SEO/GEO: research -> create -> audit -> track) as needed.
- **`--deep`** (absorbs the former `/aaron-marketing:max`): run only when the user explicitly asks for maximum-depth, exhaustive, or stress-test work. First output a phase plan, success criteria, evidence inventory, risk gates, permission checkpoints, and stop condition. Phase order for SEO/GEO goals: preflight -> research -> audit (tech / quality / visibility / authority) -> optional memory -> create (brief / write / series) -> publish gate (`/aaron-marketing:seo-geo --mode create --publish`) -> track; for influencer/ad/email/launch/social goals, coordinate the `/aaron-marketing:influencer` / `/aaron-marketing:ad` / `/aaron-marketing:email` / `/aaron-marketing:launch` / `/aaron-marketing:social` phases without bypassing their gates (launch-day-conductor never runs without a launch-readiness-auditor SHIP; social-calendar-builder's publish step never runs without a social-quality-auditor pre-publish SHIP). Do not enable `--deep` for ordinary broad work.
- Apply scenario-library risk gates: publish readiness, YMYL, schema factuality, batch scale, external side effects, memory/entity writes, reputation ethics, GEO visibility claims, insufficient data, technical indexation.
- Respect specialist boundaries: batch content stays chunked; audit/publish readiness require full veto-aware evidence; GEO visibility cannot promise citations; memory cleanup stays with memory-management; canonical entity writes stay with entity-optimizer.
- Do not route repository maintenance, version sync, release, permission, or governance requests through `/aaron-marketing:auto`; they fall outside the pack-local marketing workflow.
- Words such as apply, commit, release, publish, or fix are not automatic permission grants. Publish-package and readiness work routes through `/aaron-marketing:seo-geo --mode create --publish`; actual CMS/external publication, commits, releases, and repo edits require explicit confirmation.
- If input starts with old `/seo:*`, preserve arguments and return one copyable replacement: `audit-page -> /aaron-marketing:seo-geo --mode audit`, `audit-domain -> /aaron-marketing:seo-geo --mode audit --authority`, `check-technical -> /aaron-marketing:seo-geo --mode audit --tech`, `generate-schema -> /aaron-marketing:seo-geo --mode create --schema`, `keyword-research -> /aaron-marketing:seo-geo --mode research`, `optimize-meta -> /aaron-marketing:seo-geo --mode create --meta`, `report -> /aaron-marketing:seo-geo --mode track --report`, `setup-alert -> /aaron-marketing:seo-geo --mode track --alert`, `write-content -> /aaron-marketing:seo-geo --mode create`. Old `/aaron-seo-geo:*` names map to `/aaron-marketing:seo-geo --mode <name>` for research/create/audit/track (flags preserved); other names map 1:1 to `/aaron-marketing:*`.
- Return an execution summary by default: completed steps, evidence used, blockers or gate stops, artifacts produced, assumptions, and next safe action. Diagnostic route trace appears only when explicitly requested.
- Never write files unless explicitly requested and supported by the runtime.

## Output

Return inline artifacts by default. Files may be written only when the user explicitly asks and the runtime can write.
